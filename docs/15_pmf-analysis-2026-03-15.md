## Executive Summary

**結論: Orbit は PMF に到達していない。最大の構造的問題は「無料で価値が完結してしまう」こと。**

7人のエリートエージェントによる調査の結果、以下が判明:

- Sean Ellis テスト: **5人全員が "Somewhat disappointed"**（PMF達成には40%以上が "Very disappointed" 必要）
- 有料課金意向: **5人全員が「現状では払わない」**
- 最も評価された機能: `scan -g` による CLAUDE.md/.cursorrules 自動生成（しかし無料）
- 全員が求めた機能: **カスタムルール/プロジェクト固有の規約管理**
- 全員が不要とした機能: **Web UI のタスク管理**

## 3. 核心的な洞察

### 3.1 現在の Orbit が「ビタミン剤」な理由

開発者の痛みは「プロジェクト構造がわからない」ではない。IDE が無料で提供する。

開発者が本当に困っているのは:
- **「AI が自分のプロジェクトの"流儀"を知らない」**
- **「チームメンバーごとに AI の使い方がバラバラ」**
- **「コンテキストファイルの更新を忘れる → 古い情報で AI が回答する」**

### 3.2 「鎮痛剤」になるための条件

| 機能 | 痛みのレベル | 現状 |
|------|------------|------|
| CLAUDE.md 自動生成 | ビタミン剤 | 実装済み（無料） |
| **カスタムルール管理** | **鎮痛剤** | **未実装** |
| **手動セクション保護** | **鎮痛剤** | **未実装** |
| マルチフォーマット出力 | ビタミン剤 | 実装済み（無料） |
| Import Graph | ビタミン剤 | 実装済み（無料） |
| コンテキスト履歴・差分 | ビタミン剤 | 実装済み（有料） |
| タスク管理 | 不要 | 実装済み |

### 3.3 収益化の構造的問題

現在: **価値の90%が無料層に集中**
```
Free:  scan -g（全フォーマット）+ MCP + watch = 開発者の主要ニーズ
Pro:   履歴・差分・Smart Context = nice-to-have
Team:  組織管理 = 個人PMFなしにTeamは売れない
```

推奨: **価値を再配分**
```
Free:  scan -g（CLAUDE.md のみ）= 入口
Pro:   全フォーマット + カスタムルール + MCP（認証付き）= 日常のワークフロー
Team:  共有ルール + チーム同期 + 監査 = チーム必須
```

## 6. 最終メッセージ（7人のエージェントの総意）

**Orbit の技術基盤は強固。方向性も正しい。しかし「無料で満足」→「お金を払ってでも手放せない」の壁を越えていない。**

越えるための鍵は **Custom Rules Engine**:
- 自動検出（構造情報）= 無料 = 入口
- 人間の暗黙知（規約・ルール）= 有料 = ロックイン
- チーム共有（全員のAIに同じルール適用）= Team課金の動機

これが実装されれば、鈴木（テックリード）の「チーム全員のAI活用品質がバラバラ」という痛みを直接解決し、$152/月の Team 課金が正当化される。

**「構造の地図」ではなく「チームの流儀を AI に教える唯一のツール」へ。**
